# S0 阶段操作手册：天津监管平台对接前置准备

> 隶属于 [tianjin_supervision_plan.md](tianjin_supervision_plan.md) 的 S0 阶段展开。
> 背景：我院 **2023 年已取得互联网医院牌照**，此前由外包商（多租户 HIS SaaS，监管 SDK 供应商为纳里健康/ngari）代运营并可能一直以我院名义向监管平台上报。本次自研替换，S0 的核心不是"新申请入驻"，而是**收回账号控制权 → 摸清存量上报现状 → 拿到测试密钥 → 打通第一个请求**。
>
> ✅ 已提前完成的部分：官方 SDK jar 已到手并完成逆向对拍，**SM4/SM3/签名规则全部确认**（见 [tianjin_gateway_protocol.md](tianjin_gateway_protocol.md)，含黄金测试向量）；ICD-10 编码表、填报说明等资料已归档至 [docs/specs/tianjin/](specs/tianjin/)。原计划里"拿 SDK 对拍"这一最大不确定项已消除。

## T1 · 监管平台账号收权（第一优先级）

机构填报端入口：**https://imssp.wsjk.tj.gov.cn/hosmit-hlwqd/home.html**（出自《互联网监管平台填报说明.doc》，已归档）。

1. 先在我方内部排查是否有账号记录：牌照申办（2023 年）时的经办人邮箱/手机号、与外包商合同附件、历史邮件里搜"监管平台/imssp/账号"。
2. 分两种情况：
   - **账号在我方手里**：登录验证，立即改密码，绑定我方负责人手机号。
   - **账号在外包商手里**：书面（邮件/函件）要求移交机构账号及"秘钥生成及管理"页面的全部凭据；同时向天津市卫健委信息中心/监管平台运维方报备机构管理员变更（凭牌照与执照可以走找回流程，不依赖外包商配合）。
3. 收权后记录：登录账号、管理员手机号、密码保管人。**先不要动"正式环境密钥重置"**——见 T7 切换窗口，贸然重置会立刻切断外包商可能仍在进行的生产上报，造成断报。
4. 顺带核查平台内已维护的机构信息：机构名称、组织机构代码、诊疗科目备案、医生备案名单是否与现状一致（这些是业务上报被校验的基础数据）。

**产出：** 我方可登录的账号 + 平台内机构档案截图存档。

## T2 · 摸底外包商存量上报现状（决定切换策略）

登录平台后重点看三处：

1. **技术对接 → 接口对接数量**：各接口历史上报量、最近上报时间。判断外包商是否仍在上报、报了哪些接口。
2. **不良事件签到记录**：规范要求每日签到。若外包商早已停报，我院存在**持续断报的合规风险**——这条要在切换计划里明确"何时由谁恢复"。
3. 抽查已上报的业务数据格式（如有查询入口）：记录外包商使用的 bussID 形态，避免我方新系统 bussID 与其冲突（我方 `order_no` 建议保持自有前缀，如 `QB` 开头）。

**产出：** 一页纸现状记录：各接口最近上报日期、是否断报、bussID 形态、结论（并行期外包是否继续上报）。

## T3 · 测试环境密钥 + IP 白名单

1. 平台内进入：**申请子系统 → 技术对接 → 秘钥生成及管理**。
2. 抄录**测试环境**：`unitId`、`appKey`、`appSecret`（应为 32 位 hex，若不是先停下问平台）、测试网关地址（形如 `http://imssp.wsjk.tj.gov.cn/net-diag-service/test-openapi/api`）。
3. 申请所需**服务权限**：勾选/申请第三节矩阵中的 9 个必接方法（uploadDrugCatalogue、uploadFile、uploadConsultIndicators、uploadReferralIndicators、uploadElectMedicalRecord、uploadRecipeIndicators、uploadRecipeVerificationIndicators、uploadBusinessInfoAfter、pushMedicalDispute）。
4. IP 白名单：加入**生产服务器出口 IP**（Ubuntu 生产机）与联调期开发出口 IP。注意家庭/办公宽带是动态 IP，联调建议直接在生产机或固定 IP 的跳板机上跑。
5. 密钥保管：只进服务器 `.env`（`TJ_APP_KEY/TJ_APP_SECRET/TJ_UNIT_ID/TJ_GATEWAY_URL`），不进 git、不进聊天记录。

**产出：** `.env.tj.example` 模板 + 密钥已配置到测试机。

## T4 · 首次连通性验证（拿到密钥当天即可做）

已备好脚本 [backend/scripts/tj_ping.py](../backend/scripts/tj_ping.py)：

```bash
cd backend && pip install gmssl
# 编辑脚本头部 5 个配置项（网关/appKey/appSecret/unitId/organID）
python scripts/tj_ping.py
```

脚本先跑**本地国密自检**（与官方 SDK 黄金向量逐字节对拍，不通过不会发网络请求），再向测试网关推 1 条演示药品（`useFlag=2` 取消态，不污染目录）。

结果判读：

| 返回 | 含义 | 处理 |
| :-- | :-- | :-- |
| `body.msgCode=200` | 全链路通 | ✅ T4 完成 |
| `code=40007` | IP 不在白名单 | 回 T3 补白名单 |
| `code=40004/40010` | 密钥/签名错 | 核对 appSecret 是否 32 hex；截图问平台 |
| `code=40006` | 服务权限未开 | 回 T3 申请方法权限 |
| `body.msgCode=-99` | 链路已通，字段缺失 | 视同通过（msg 会列缺失字段） |

**产出：** 请求/响应截图归档。

## T5 · 口径确认清单（发平台对接人，要书面回复）

以下问题直接决定 S2/S3 的开发形态，逐条要到明确答复（规范修订记录中的平台侧作者可作为线索，或通过 T1 报备时建立的联系渠道）：

1. **业务归类**：我院业务为图文咨询 + 视频问诊（复诊）。确认：图文按 2.2.1 在线咨询（consultationType=1，onsultationAttribute=1 诊疗咨询）、视频按 2.2.2 在线复诊 + 2.2.10 电子病历上报，是否认可？
2. **2.2.9 预约挂号接口**：我院无线下门诊、"挂号"即复诊预约，此接口是否需要对接？
3. **测试达标标准**：每个接口需要多少条测试数据算达标？达标后由谁确认可切正式？
4. **上报时间窗**：正式环境每日推送有无时间要求？（我方计划每日 01:30 推前一日终态数据。）
5. **处方核销**：我院有药房自取（deliveryType=0）与物流配送（=1）两种；处方超效期未取药按 `deliveryType=4 自动失效` 上报是否正确？
6. **存量与幂等**：切换前由外包商上报的历史数据是否需要我方补报？同一 unitID 下 bussID 是否按唯一键幂等（重复上报会覆盖还是报错）？
7. **正式密钥重置**：重置后旧密钥是否立即失效？可否指定生效时间（配合切换窗口）？
8. **附件上传**：uploadFile 的单文件大小、类型限制；附件 id 有效期。
9. **依赖关系**：处方明细 drcode 是否必须先在药品目录备案成功后才能上报处方？（决定 S3 上报顺序。）
10. **评价时序**：业务数据 T+1 上报，当天产生的评价随次日批次一起传是否可以（规范说"须先完成业务数据推送否则拒收"）？

**产出：** 问题-答复对照表，归档到 `docs/specs/tianjin/`。

## T6 · 字典对照表（医务/药房参与，与 T5 并行）

做四份 CSV（建议放 `backend/data/tj_dicts/`，S2 写导入脚本直接消费）：

1. **医生 → 诊疗科目编码**（规范 3.1）：每位在册医生映射 `subjectCode/subjectName`，如呼吸内科医生 → `03.01 呼吸内科专业`；同时确认每人身份证号收集渠道（S2 要加密入库）。
2. **科室 → 科室类别编码**（规范 3.2）：现 `doctors.dept` 是自由文本（如"呼吸内科"），映射到大类编码（如 `03 内科`）；`deptName` 保留具体科室名。
3. **药品 → 监管分类 + 国家药品编码**（规范 3.10）：现有 `drugs` 表逐条映射 `drugClass`（如 010102 头孢菌素类）、`countrydrcode`（国家药品编码，药房从药品说明书/医保编码库查）、补包装规格与产地。**这份最费人工，尽早给药房布置。**
4. **ICD-10 导入源**：已归档 `国家临床版2.0疾病诊断编码（ICD-10）.xlsx`（约 3 万条）与中医版；S2 的 `scripts/import_icd10.py` 直接读它建 `icd10_codes` 表，无需再加工。

**产出：** 四份 CSV/XLSX + 每位医生的身份证收集完成。

## T7 · 机构信息核对与切换窗口计划

1. **organID**：从《医疗机构执业许可证》/营业执照确认全国统一的组织机构代码（规范字段 Varchar(20)），与平台机构档案里登记的一致；`ORGAN_NAME` 用登记全称。
2. **切换窗口方案**（写成一页纸，S5 执行）：
   - 并行期：外包商维持正式环境上报（尤其不良事件签到）不动；
   - 我方测试环境达标（S5 前置）；
   - 选定切换日 D：D-1 书面通知外包商停止上报 → D 日 00:00 起我方接管每日批次与签到 → D 日重置正式密钥并更新白名单（旧密钥失效，外包商物理断开）；
   - 回退预案：我方连续 2 天上报失败则临时恢复外包商密钥（因此 D 后一周内先别在平台删除外包商 IP）。

**产出：** 切换窗口一页纸，负责人签字。
