# 逑贝互联网医院 · 系统实施路线图（Roadmap）

> 本文是全系统的实施总纲。四个工程（患者端小程序、医生端小程序、`backend` 中央服务端、`admin-web` 运营后台）按本文的里程碑推进。配套文档：[FRD](mini_program_prd.md) · [小程序开发指南](mini_program_dev_guide.md) · [后端 PRD](backend_saas_prd.md) · [后端开发指南](backend_saas_guide.md) · [全链路方案](qiubei_hospital_full_solutuion.md)。

---

## 一、总体策略

1. **MVP 闭环优先。** 第一阶段只追一条线：**挂号付钱 → 医生开关接诊叫号 → 跨端 WebSocket 唤醒 → 双端视频 → 开方 → 药费 → 钱进商户号**。其余功能一律延后。
2. **纵向切片（Vertical Slice）。** 不要"先把后端全写完再联调"。每个里程碑都是**端到端一条线**：后端接口 → 小程序换掉对应 mock → 真机验证通过，再做下一片。
3. **后端在前，小程序紧随。** 小程序前端已完成（切图 + 逻辑 mock），其进度已被后端阻塞；因此关键路径是后端，小程序做"对接联调"。
4. **合规与外围并行起跑。** 资质、CA、卫健委接口、云服务器、正式小程序注册周期长，**从 M1 就并行启动**（见第五节），否则会成为上线瓶颈。
5. **PC 运营后台按需跟进。** 仅"药师审方"在 MVP 处方链路上（M5 做最小版），其余运营功能放到增强阶段。

## 四、里程碑详解

> 每个里程碑统一格式：**目标 → 后端任务 → 小程序任务 →（PC 任务）→ 验收标准 → 前置依赖**。验收通过才进入下一里程碑。

### M1 · 账号与准入（登录闭环）
- **目标：** 两端用真实账号登录，打通登录守卫与准入闸门。
- **后端：** JWT 登录（手机号 code 换 token）；`doctors` 白名单校验；就诊人实名认证（二要素 + 加密存储）；知情同意存证（`consents`）；RBAC 依赖。
- **小程序：** 两端 `login` 接真接口、`app.js` 守卫换真 token；患者端就诊人管理实名表单；首次问诊知情同意签署回传。
- **验收：** 未登录访问受保护页被拦截并登录成功；医生非白名单被拒；就诊人实名落库（脱敏返回）。
- **依赖：** 微信小程序 `getPhoneNumber` 权限（需企业主体/正式 AppID 才完整，可先测试号模拟）。

### M2 · 挂号支付（资金第一环）
- **目标：** 跑通"选医生 → 下单 → 微信支付成功 → 候诊中"。
- **后端：** 排班/号源表 + Redis 号源锁（`DECRBY` 防超卖）；医生列表/排班 API；创建订单（`0 PENDING`）；**微信支付下单（五元组）**；**支付回调状态机 `0→1`（行锁 + 幂等）**；15 分钟超时未付自动 `0→9`（Redis 键空间通知）。
- **小程序：** 页面2 医生列表、页面3 详情/排班 接真数据；`utils/pay.js` 换真实下单；支付成功回首页挂排队条（读 `status=1`）。
- **验收：** 真机小额真实支付成功 → 订单变 `WAITING` → 进 Redis 排队队列；超时订单自动取消。
- **依赖：** **微信支付商户号**（需正式主体，C 角尽快办）；可先用沙箱/小额验证。

### M3 · 接诊与信令（跨端唤醒）
- **目标：** 医生接诊 → 患者任意页面秒级被唤醒。
- **后端：** WebSocket 信令中心（`/ws`，在线态 + `room:status`）；`INIT_CALL → CALL_INVITE → CALL_ANSWER → START_STREAM`；4s 心跳、30s 断线保护；接诊锁 `1→2` + 医生置忙；候诊队列实时 API。
- **小程序：** 医生端 D1 开关/队列/立即接诊 接真接口；患者端 `app.js` 挂全局 `CALL_INVITE` 监听 → 拉起 `call` 页；`ANSWER → redirectTo video-room`。
- **验收：** 医生点"立即接诊"，患者无论停在哪页都被拉起接听页；拒绝/接听信令双端一致。
- **依赖：** M1（鉴权）、M2（订单进入 WAITING）。

### M4 · 双向视频（RTC）
- **目标：** 双端真机视频画面互通。
- **后端：** RTC `UserSig` 正式算法（腾讯云 TRTC 官方算法替换骨架）；房间号下发。
- **小程序：** `video-room` / D2 的 `live-player`/`live-pusher` 填真实推拉流、大小窗对调；进房前音视频权限校验。
- **验收：** 两部真机进同一房间，医生看患者、患者看医生，挂断 `CALL_FINISHED` 双端联动。
- **依赖：** **腾讯云 TRTC 开通 + 小程序实时音视频类目**（有审核周期，C 角提前办）。

### M5 · 开方与药师审方
- **目标：** 开方 → 药师审核 → 患者看到带章处方。
- **后端：** EMR 病历（主诉/现病史/诊断 ICD-10/医嘱，非空校验）；开方提交 `2→3`；药品字典（**特殊限售药拦截**）；药师审方 `3→5`（通过）/`3→4`（驳回，必填理由）→ 驳回回 `4→3`；处方 PDF + CA 加签（**M5 可先用占位章，M9 接正式 CA**）。
- **PC（admin-web）：** 药师审方页接真接口（通过/驳回）——MVP 必需的最小运营功能。
- **小程序：** 医生端 D2 病历/开方提交；患者端页面6 处方展示；医生端 D1 驳回重开区。
- **验收：** 医生开方 → 药师在 PC 后台通过 → 患者页面6 显示带章处方；驳回能回流医生重开。
- **依赖：** M3/M4（问诊中）。

### M6 · 药费与分账（MVP 闭环达成）✅
- **目标：** 药费支付成功，资金按规则进商户号并分账。
- **后端：** 药费支付下单 + 回调 `5→6`；财务分账写 `ledger`（医院/医生/平台，分为单位整数）；订单归档"待药房配送"。（提现审批可延到 M8）
- **小程序：** 页面6"去支付药费"接真支付；成功跳个人中心、订单归档。
- **验收（MVP 总验收）：** 一笔真实订单完整跑通 **挂号付钱 → 接诊叫号 → 跨端唤醒 → 双端视频 → 开方审方 → 药费 → 资金进天津主体商户号并完成分账**。
- **依赖：** M2（支付通道）、M5（处方生效）。

> 🎯 **M6 达成 = 第一阶段目标完成。** 之后是增强与合规上线。

### M7 · 履约与通知
- **后端：** 订单履约（配药/发货/物流/签收时间轴）；消息通知中心（微信**订阅消息**：呼叫/处方/药费/物流/退款）；退款 `1/2→7`、`5→7`。
- **小程序：** 页面10 物流追踪、页面8 消息诊室接真数据；退款入口。
- **验收：** 关键节点推送订阅消息；物流时间轴可查；异常订单可退款。

### M8 · 运营后台完善（admin-web）
- **后端 + PC：** 医生入驻/多点执业资质终审（OSS 防盗链）；药品字典管理；财务对账 + **提现审批**（商家转账到零钱状态机）；监管上报监控面板（成功率/失败/一键重报）；RBAC 菜单。
- **验收：** 行政能终审医生、药师能审方、财务能审提现并打款、运营能看监管面板。

### M9 · 合规网关硬化（命门）
- **后端：** 天津卫健委实时上报（AES-128-CBC/SM4 + 动态 Sign，Celery 异步 + 指数退避 + 死信队列）；CA 机构正式对接（SM2 云端静默加签，替换占位章）；音视频录制 + 处方 PDF **归档存储 15 年**（Lifecycle）。
- **验收：** 接诊/审方触发的上报异步成功、失败可重报；处方含合法 CA 签章；档案进归档存储且不可删。
- **依赖：** 卫健委接口规范 + CA 合同（C 角在 M1 就启动）。

### M10 · 提审上线
- **运维/前端：** 线上 HTTPS 业务域名 + WSS 信令、ICP 备案；关闭"不校验合法域名"；**两个小程序分别上传、分别提审**（医疗类目备齐资质）；小流量灰度 → 全量；**等保三级自查**（VPC 隔离、密钥不硬编码、审计日志、加密存储、限流）。
- **验收：** 两端审核通过、线上全链路正常、首单真实问诊闭环成功。

## 七、联调与环境约定

- **后端地址：** 小程序 `app.globalData.baseUrl`/`wsUrl` 联调期指向局域网 IP（如 `http://192.168.x.x:8000`、`ws://192.168.x.x:8000/ws`），上线改线上 HTTPS/WSS。
- **域名校验：** 联调期微信开发者工具勾选"不校验合法域名"；上线前务必关闭并登记合法域名。
- **契约来源：** 一律以后端 Swagger（`/docs`）为准，字段命名前后端对齐，避免二次返工。
- **状态机单一真相：** 所有状态变更只能经后端 `order_service.transition`，前端只读展示，严禁前端改 `status`。
- **数据库迁移：** 表结构用 Alembic 管理，禁止手改线上库。

## 九、MVP 闭环定义（第一阶段验收红线）

> 当且仅当下面这条链路在**两部真机 + 真实小额支付**下全程跑通，视为 MVP（M1–M6）完成：

**患者登录 → 选医生挂号 → 微信支付成功（订单 WAITING）→ 医生端开诊接诊（CONSULTING）→ 患者被 WebSocket 唤醒接听 → 双端 TRTC 视频 → 医生开方（AUDITING）→ 药师审核通过（PRESCRIBED）→ 患者支付药费（FINISHED）→ 资金进天津主体商户号并完成分账。**
