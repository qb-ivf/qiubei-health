# 逑贝互联网医院——前端微信小程序详细需求与交互设计文档 (FRD)

## 一、 全局技术规范与性能指标

### 1.1 双端独立小程序与分包加载策略

> ⚠️ **架构基线（与《全链路合规方案》对齐）：** 为满足微信医疗类目审核红线、明确医疗责任主体，**患者端与医生端是两个物理隔离的独立小程序**（独立 AppID、独立 codebase、独立审核与发布），并非同一小程序的两个分包。两端仅通过统一中央服务端（HTTPS 业务 / WSS 信令）通信。

| 小程序 | 暂定名 | 承载页面 | 体积约束 |
| :--- | :--- | :--- | :--- |
| **患者端**（小程序 A） | 逑贝健康 | 页面 1~7 | 主包 ≤ 2MB |
| **医生端**（小程序 B） | 逑贝医生端 | 页面 D1~D3 | 主包 ≤ 2MB |

由于互联网医院功能复杂、未来还会扩模块，**每个小程序各自**从第一天起采用分包加载（Subpackaging）架构：

* **患者端 · 主包：** 仅含【页面 1：首页】、【页面 7：个人中心】、全局 4-Tab 栏与公共样式。
* **患者端 · 分包 A（问诊流）：** 页面 2、3、4、5、6（涉及 RTC 音视频与支付，按需加载）。
* **医生端 · 主包：** 仅含【页面 D1：接诊大厅】、【页面 D3：我的工作台】与公共样式。
* **医生端 · 分包 B（开方业务）：** 页面 D2（视频开方一体化，涉及 RTC，按需加载）。

### 1.2 全局权限与状态劫持机制
* **登录拦截：** 在 `app.js` 的 `onLaunch` 生命周期中，封装全局路由守卫。未登录（本地 Storage 查不到 Valid Token）的用户进入页面 2、3、D1、D2 时，前端硬性重定向弹出统一的微信手机号一键授权登录弹窗。
* **音视频权限检测：** 进入页面 4、5、D2 前，必须调用 `wx.getSetting` 校验用户是否授权了 `scope.camera`（摄像头）和 `scope.record`（麦克风）。若拒绝，前端展示友好的全屏引导页，指引点击右上角“...”进入设置手动开启。

### 1.3 订单状态机前端映射（与服务端枚举严格对齐）
状态由中央服务端封闭式管控，**前端只读取、只展示、无权擅改**。各页面的状态展示必须直接映射后端 `status` 枚举，避免口径不一：

| `status` | 含义 | 前端典型呈现位置 |
| :-: | :--- | :--- |
| `0 PENDING` | 待支付 | 页面3 支付前、页面7「待支付」格（含 15 分钟倒计时） |
| `1 WAITING` | 候诊中 | 首页黄色置顶排队条、页面7「待接诊」红点、D1 候诊队列 |
| `2 CONSULTING` | 问诊中 | 页面5 视频中、页面7「问诊中」 |
| `3 AUDITING` | 待药师审核 | 页面6 头部「处方审核中」、页面7 |
| `4 REJECTED` | 处方被驳回 | 医生端 D1 顶部「待重开处方」红点（见 §4 驳回回路） |
| `5 PRESCRIBED` | 已开方/待付药费 | 页面6 可支付、页面7「处方药费待支付」 |
| `6 FINISHED` | 问诊完成 | 页面7「已完成」、物流追踪入口 |
| `7 REFUNDED` | 已退款 | 页面7、退款结果提示 |
| `9 CANCELLED` | 已取消（超时未付） | 页面3 倒计时归零、页面7 灰显 |

### 1.4 设计规范 Token 与适老化（无障碍）
* **主色定义（统一口径）：** 品牌主色 **医疗蓝 `#2A76F4`**（导航、主按钮、链接）；**视频问诊金刚区**使用**强调绿 `#07C160`**（微信支付绿系，传达"在线/可接听"），二者分工明确，勿混用。辅助色：警示橙 `#FF8F1F`（费用/即将约满）、错误红 `#FA5151`、成功绿 `#07C160`。
* **圆角/阴影：** 卡片统一 `12px` 圆角 + 微弱阴影 `0 2px 8px rgba(0,0,0,.06)`。
* **适老化：** 正文最小字号 ≥ `32rpx`，核心操作按钮高度 ≥ `88rpx`、可点区域 ≥ `48px`；色彩对比度满足 WCAG AA（≥ 4.5:1）；关键文案避免纯灰低对比。
* **就诊人切换** 等高频组件需支持 TalkBack/无障碍朗读 `aria-role`。

### 1.5 性能指标（量化验收基线）
* **包体积：** 各端主包 ≤ 2MB，单分包 ≤ 2MB，整包 ≤ 20MB。
* **首屏：** 首页可交互（TTI）≤ 2s（4G）；页面切换 ≤ 500ms。
* **音视频：** 接听到视频首帧 ≤ 2s；卡顿率（500ms 无帧）< 5%。
* **长连接：** WebSocket 心跳 4s/次，断线 30s 内自动重连（见 §1.6）。

### 1.6 全局异常 / 空 / 加载态规范
所有数据页面必须显式设计三态，禁止白屏：
* **加载态：** 列表/卡片用骨架屏（Skeleton），网络请求 > 300ms 才显 loading，避免闪烁。
* **空态：** 医生列表、处方档案、候诊队列等为空时展示插画 + 引导文案 + 重试/返回按钮。
* **异常态：** 接口失败统一兜底页（"网络开小差，点击重试"）；
  * **弱网/断线重连：** 视频页（页面5/D2）监听长连接，断开后展示半透明"网络不稳定，正在重连…（剩 N 秒）"浮层，对齐服务端 12s 无响应→30s 重连保护期；30s 仍失败则提示"通话已断开"并按 `7 REFUNDED` 走退款引导。
  * **支付异常：** `wx.requestPayment` 的 `fail` 需区分用户主动取消（静默）与真实失败（提示重试）。

## 二、 全局准入与合规流程（首诊前置，硬性卡点）

互联网诊疗受卫健委强监管，以下流程是**问诊前不可绕过的合规闸门**，前端必须实现。

### 2.1 登录授权流程
* 触发时机见 §1.2 路由守卫。统一弹出**微信手机号一键授权**组件（`button open-type="getPhoneNumber"`），获取加密手机号交后端换取 JWT。
* 医生端额外校验：手机号须在后端 `doctors` 白名单内，非白名单一律拦截并提示"非认证执业医师"。
* Token 失效（401）时全局拦截，清本地 Storage 并重新拉起登录。

### 2.2 就诊人实名认证与管理
* **实名制要求：** 添加就诊人须录入真实姓名 + 身份证号（前端做格式校验，加密后交后端核验），自费/医保信息按需。
* 支持多就诊人（家庭成员）管理：增/删/改、设默认就诊人；首页与页面3 的就诊人切换均读此列表。
* 为儿童等无身份证场景预留"监护人代办"关系字段。

### 2.3 互联网诊疗知情同意与隐私协议
* **首次问诊强制签署：** 在页面3 支付前或首次进入问诊流时，强制弹出《互联网诊疗知情同意书》《隐私政策》《医疗风险告知》，用户须**主动勾选 + 点击同意**方可继续，签署结果（含时间戳）上报后端留痕。
* 未签署不得进入支付与音视频环节；协议入口常驻页面7「隐私服务协议」可随时复看。

## 三、 患者端小程序核心页面详细需求（7 个核心页面 + 3 个支撑页）

> 说明：页面 1~7 为 MVP 核心闭环；页面 8（消息诊室）、9（搜索）、10（订单物流）为 Tab/引用补全页，可在核心闭环跑通后排期。

### 页面 1：小程序首页
* **UI 布局：**
  * **顶部：** 【自定义 NavigationBar】。左侧显示当前默认就诊人姓名（如：“张三 ＞”），点击唤起半透明底部抽屉（ActionSheet）切换就诊人；右侧保持微信原生胶囊按钮。
  * **中部：** 轮播图（Banner） + 搜索框（支持点击跳转搜索页）。
  * **核心区：** 金刚区大卡片。采用“2*2”网格，左上角为“视频问诊”核心大卡片，采用品牌色（高亮绿色）背景，带“医生在线视频秒接听”的小标签提示。
* **交互逻辑：**
  * 点击“视频问诊”卡片，前端触发 `wx.navigateTo` 路由，跳转至【页面 2：医生列表页】。
  * 底部常驻全局 Tab 栏（4个）：[首页]（当前激活）、[科室挂号]、[消息诊室]、[我的]。

### 页面 2：医生列表页
* **UI 布局：**
  * **顶部固定（Sticky）：** 科室横向滚动滑动条（Scroll-View），如：内科、儿科、妇科、皮肤科。
  * **二级固定：** 排班日期滑动横条（展示未来 7 天的日期、星期几、以及“今日有号”的 Badge 气泡）。
  * **下方：** 医生卡片垂直流（Scroll-View，支持触底加载更多 `onReachBottom`）。
* **组件细节：** 医生卡片包含：医生头像（圆形，50dp）、姓名、职称（如：主任医师）、擅长主诉文本（超出 2 行自动省略号 `text-overflow: ellipsis`）、挂号费（高亮橙色，如：¥30.00）。右侧常驻一个胶囊按钮，文字为“预约”。
* **交互逻辑：** 点击卡片任意区域或“预约”按钮，携带 `doctor_id` 参数跳转至【页面 3】。

### 页面 3：医生详情与挂号确认页
* **UI 布局：**
  * **上半部：** 医生立体化名片，展示详细履历、好评率、资质证书荣誉。
  * **中部：** 【就诊人选择组件】。默认展示当前患者，右侧带“修改/添加”按钮。
  * **下半部：** 排班时间段抽屉。展示该医生选定日期的具体出诊时段（如：09:00-09:30、09:30-10:00），剩余号源数小于 2 时，号源标签显示红色“即将约满”。
  * **底部固定栏（Fixed）：** 左侧显示“合计：¥30.00”，右侧为宽度 50% 铺满的“立即支付”大按钮。
* **交互逻辑：**
  1. 点击“立即支付”，前端向后端 API 发起下单请求，获取微信支付五元组参数（`timeStamp`, `nonceStr`, `package`, `signType`, `paySign`）。
  2. 前端调用微信原生支付组件 `wx.requestPayment`。
  3. **支付成功回调（success）：** 弹出原生 Toast 提示“预约成功，正在为您安排诊室”，延时 1.5 秒后，调用 `wx.switchTab` 强制返回【页面 1：小程序首页】，并在首页顶部挂载一个“您有正在排队的视频问诊，请保持手机亮屏”的全局置顶黄色提示条。

### 页面 4：远程视频呼叫与接听页
* **UI 布局：**
  * 全屏无导航栏样式（`"navigationStyle": "custom"`），背景采用高斯模糊/毛玻璃效果（`backdrop-filter: blur(20px)`），中央渲染呼叫医生的圆形大头像（带向外扩散的绿色水波纹/涟漪 CSS 动画）。
  * 头像下方显示：`“XXX 主任医师 正在邀请您进行视频问诊...”`。
  * 底部 20% 高度处，并排两个大圆形图标按钮：左侧红底白图标（拒绝），右侧绿底白图标（接听，带放大缩小呼吸动画）。
* **技术与交互逻辑：**
  * **全局长连接唤醒：** 前端在 `app.js` 的 WebSocket 监听器中，一旦收到 `type: "CALL_INVITE"` 信令，不论用户当前停留在哪个页面，立刻强行执行 `wx.navigateTo` 压栈拉起本页面。
  * 点击“拒绝”：向后端发送 `CALL_REJECT` 信令，清空本地响铃状态，调用 `wx.navigateBack` 退出。
  * 点击“接听”：向后端发送 `CALL_ANSWER` 信令，并使用 `wx.redirectTo`（不留历史栈）直接杀掉本页，进入【页面 5：全屏双向视频诊室页】。

### 页面 5：全屏双向视频诊室页
* **UI 布局与原生组件配置：**
  * **大窗（背景全屏）：** 微信原生 `<live-player>` 组件，用于渲染医生画面。属性必须配置：`src="后端返回的拉流地址/RoomID"`, `mode="RTC"`, `autoplay="true"`, `sound-mode="speaker"`, `object-fit="fillCrop"`（填满全屏）。
  * **小窗（右上角悬浮）：** 微信原生 `<live-pusher>` 组件，本地摄像头预览。宽高固定为 120dp * 160dp，四周带圆角及阴影。属性配置：`src="后端返回的推流地址"`, `mode="RTC"`, `autopush="true"`, `enable-camera="true"`, `enable-mic="true"`。
  * **控制层（底部半透明浮窗）：** 麦克风开关（Mute）、摄像头翻转（Flip）、全屏挂断按钮（居中，加大红底圆形）。
* **核心交互：**
  * **大小窗对调：** 点击右上角小窗，触发前端 `Page` 数据的 `data.isSwapped` 切换，无缝对调 `<live-player>` 和 `<live-pusher>` 的布局层级（Z-Index）和宽高，允许患者大窗看自己，小窗看医生。
  * **被动断开监听：** 前端持续监听 WebSocket 信令。一旦收到后端或医生端下发的 `CALL_FINISHED`（医生开完处方断开视频），前端立刻弹窗提示“通话已结束，正在为您生成电子处方...”，并在 1 秒后自动跳转至【页面 6：诊后电子处方与药费支付页】。

### 页面 6：诊后电子处方与药费支付页
* **UI 布局：**
  * **顶部：** 绿色成功状态 Banner，文字：`“本次远程问诊已结束，医生已为您开具临床电子处方”`。
  * **中部：** 标准 Rp 处方卡片。采用白色实体卡片样式，上方居中印刷“逑贝互联网医院电子处方单”字样；中部排列药品表格（药名、规格、数量、单价、用法用量如：一日三次，一次一粒，饭后服用）；右下角使用绝对定位堆叠一张半透明的红色“互联网医院处方专用章”医疗专用电子公章印章图片。
  * **底部固定栏：** 左侧显示药费总计，右侧绿色大按钮：`“去支付药费（送药上门）”`。
* **交互逻辑：**
  * 为了防止处方作废或重复支付，点击“去支付药费”同样走微信原生支付流。
  * 支付成功后，跳转至【页面 7：个人中心】，并将该订单状态归档为“待药房配送”。

### 页面 7：个人中心/“我的”页面
* **UI 布局：**
  * **顶部：** 高档品牌色背景，展示患者微信头像、昵称，下方展示电子健康卡/就诊人多卡切换滑块。
  * **中部：** 【问诊状态网格】。包含 4 个图文入口：待接诊（动态带数字红点 `wx.showTabBarRedDot`）、问诊中、处方药费待支付、已完成。
  * **下半部：** 功能列表行（List Row）。[我的处方档案]、[挂号预约记录]、[就诊人管理]、[隐私服务协议]。

### 页面 8：消息诊室（全局 Tab「消息诊室」）
> 全局 4-Tab 之一，原指南缺失，此处补全。
* **UI 布局：** 会话列表。区分**系统通知**（挂号成功、处方已开、药费待付、物流更新、退款结果）与**问诊会话**（按医生/订单聚合）。每行带未读红点。
* **交互逻辑：** 点击系统通知跳对应页（如"处方已开"→页面6）；点击问诊会话进入该订单的图文/状态详情。会话状态徽标直接映射 §1.3 状态机。

### 页面 9：搜索页
> 页面1 搜索框跳转目标，原指南缺失，此处补全。
* **UI 布局：** 顶部搜索输入框（聚焦自动唤起键盘）、历史搜索标签、热门科室/医生推荐。
* **交互逻辑：** 输入关键词实时联想（医生姓名 / 科室 / 主诉病症），点击结果按类型跳转页面2（列表）或页面3（医生详情）。空结果展示空态（§1.6）。

### 页面 10：订单与物流追踪页
> 页面6「送药上门」与个人中心「已完成」的承接页，原指南缺失，此处补全。
* **UI 布局：** 订单状态时间轴（§1.3：已付药费→药房配药→已发货→配送中→已签收）、收货地址、药品清单、电子处方查看入口、物流单号与承运商。
* **交互逻辑：** 进行中订单展示物流轨迹；支持复制单号、联系客服、申请售后/退款（触发 `7 REFUNDED` 流程）。

## 四、 医生端小程序核心页面详细需求（3 个页面）

### 页面 D1：医生首页/接诊大厅
* **UI 布局：**
  * **顶部：** 医生实名卡片（头像、姓名、科室、多点执业标示）。右侧放置一个醒目的异步滑块开关组件（`wx-switch`），文字提示为：“开启接诊状态”。
  * **中部：** 今日业绩看板（今日已接诊人数、待药师审核处方数、今日预计收益）。
  * **下半部：** 【候诊室动态排队队列】。垂直列表展示当前分配给该医生的患者信息卡片。包含：排队序号（如：001号）、患者姓名（脱敏）、性别、年龄、挂号时填写的主诉文本（超出 3 行提供“展开”按钮）。
* **核心交互与控制逻辑：**
  1. 当 `switch` 开关为 OFF（关闭） 时，下方排队队列整体呈灰色半透明置灰状态（`opacity: 0.5`），列表第一位的卡片右侧按钮显示为灰色“休治中”。
  2. 当医生将开关拨动到 ON（开启） 时，前端触发 API 更改后端 Redis 状态，排队列表瞬间激活。排在第一位的患者卡片高亮闪烁，右侧“休治中”按钮变成蓝色高亮的 “立即接诊”。
  3. 医生点击“立即接诊”，前端调用 `wx.navigateTo` 携带患者及房间参数，全速杀入【页面 D2】。
  4. **处方驳回回路（对应状态机 `4 REJECTED`）：** 候诊队列上方增设【待重开处方】提醒区（带数字红点）。药师驳回后该订单回流至此，卡片展示**药师驳回理由**（如"配伍禁忌""用量不当"）。医生点击进入 D2 的 `[开具处方]` Tab（视频可不重连，纯图文改方），修订后重新"数字签名并发送"，订单回到 `3 AUDITING` 等待药师二次审核。

### 页面 D2：视频问诊与开处方一体化页面
* **布局架构（防软键盘遮挡硬性限制）：**
  整体页面高度 100vh，采用垂直 Flex 布局（`flex-direction: column`），严禁使用全局滚动。上半部固定视频（高度 40vh），下半部固定作业面板（高度 60vh）。
* **双向施工作业面板各个 Tab 详细需求：**
  1. **`[患者主诉]` Tab：** 纯文本展示患者挂号信息，下方九宫格展示患者上传的既往病例照片，支持点击图片触发微信原生全屏预览 `wx.previewImage`。
  2. **`[书写病历]` Tab：** 提供三个标准的 `textarea` 输入框：[现病史]、[初步诊断]、[医嘱]。
     * **技术适配（规避键盘推起视频 Bug）：** 所有 `textarea` 组件必须强制配置属性 `adjust-position="false"`（禁止键盘推起页面）以及 `cursor-spacing="20"`。前端利用监听输入框聚焦 `bindfocus` 事件，动态计算软键盘高度（`e.detail.height`），将下部作业面板的底部内边距 `padding-bottom` 设为键盘高度，确保输入框内容永远浮在软键盘上方，而上部 40vh 的视频区在视觉上雷打不动，医生边看视频边打字输入病历。
  3. **`[开具处方]` Tab：** * 顶部带一个搜索框。输入药名拼音首字母，触发实时下拉联想。点击联想出的药品，该药品以卡片形式塞入下方的“已开药品列表”。
     * 药品卡片带数字加减器（Stepper 组件，修改数量）以及用法用量下拉选择器（Picker 组件，选择“一日三次，一次一粒”）。
* **底部常驻提交大按钮：** 文字为“数字签名并发送处方”。
* **交互闭环：** 医生点击“发送处方”，前端触发 CA 签名 loading 动画，成功后订单进入 `3 AUDITING`（待药师审核），页面自动清栈并强退回【页面 D1：接诊大厅】，等待叫号下一位。若此处方源自药师驳回（见 D1 第 4 点），提交后同样回到 `3 AUDITING` 等待二次审核。

### 页面 D3：个人中心/“我的工作台”
* **UI 布局：** 医疗严谨风格风格，大底色为深蓝色。展示医生职业证件照、姓名、职称、医院科室。
* **主要组件：**
  * 【执业钱包组件】：展示“可提现余额（元）”，右侧一个高亮按钮“立即提现”。点击提现弹出输入金额弹窗，调用后端分账提现接口。
  * 【配置管理列表】：
    * [出诊排班管理]：点击进入日历选择器，医生可自助勾选下周哪天、哪个时段出诊，并设置限号数。
    * [诊金设置]：可自由输入视频问诊的挂号费金额（设置上限与下限）。
    * [常用快捷词回复]：医生可预设“请问您现在体温多少度？”等常用语，在页面 D2 问诊中可以一键点击快捷输入。
