# 逑贝互联网医院 SaaS 后端管理系统——系统需求与设计文档 (PRD)

## 一、 系统全局设计原则与架构总体架构

### 1.1 技术栈规范（基于 AI 高效生成）
* **后端语言：** Python 3.10+（高并发、生态成熟，AI 代码准确率最高）
* **核心框架：** FastAPI / Django REST Framework (DRF)
* **数据库：** MySQL 8.0+（核心业务数据） + Redis 7.0+（WebSocket信令、音视频 Token 缓存、排队队列）
* **网络协议：** HTTPS（业务接口） + WSS（WebSocket 安全长连接信令）

### 1.2 网络与安全部署架构（满足等保三级刚需）
* 由于系统涉及医疗隐私及卫健委监管，必须部署在阿里云/腾讯云的**医疗专区云**。
* **网络隔离：** 核心数据库（MySQL）必须部署在私有网络（VPC）的私有子网内，严禁对公网开放。
* **网关接入：** 所有的请求必须经过 API 网关或配置了 SSL 的 Nginx 反向代理服务器，统一对外提供 HTTPS 服务。

### 1.3 接口规范与错误约定（补充）
* **版本化：** 所有接口统一前缀 `/api/v1`，向后兼容靠新增版本号，不破坏旧端。
* **统一响应：** 成功直接返回业务体；错误统一 `{ "detail": "原因" }` + 合理 HTTP 状态码（参数 422、未授权 401、无权限 403、状态冲突 409、限流 429）。
* **幂等键：** 支付回调、提现、上报等关键写操作要求携带幂等键（订单号/任务号），服务端去重。
* **接口文档：** FastAPI 自带 Swagger（`/docs`）即契约，交付前端零沟通对接。

### 1.4 数据模型总览（核心表，供 AI 建模对齐）
| 表 | 关键字段 | 说明 |
| :--- | :--- | :--- |
| `users` | role, openid, phone(加密), id_card(加密) | 患者/医生/药师/管理员统一账号 |
| `patients` | user_id, name, id_card, relation, verified | 就诊人（实名，多就诊人） |
| `doctors` | user_id, license_no, practice_no, dept, title, audit_status | 医生资质（白名单来源） |
| `schedules` / `slots` | doctor_id, date, start, end, quota | 排班模板 + 号源 |
| `orders` | order_no, patient_id, doctor_id, status, fee | 订单（状态机，status 建索引） |
| `prescriptions` | order_id, diagnosis, items, audit_status, ca_sign, pdf_url | 处方 + CA 签名 + PDF |
| `consents` | patient_id, type, signed_at, snapshot | 知情同意/隐私协议签署存证 |
| `withdrawals` | doctor_id, amount, status | 提现单 |
| `ledger` | order_id, hospital, doctor, platform | 分账流水 |
| `gov_reports` | biz_id, type, status, retries | 卫健委上报任务/重试 |
| `audit_logs` | actor, action, target, ip, at | 操作审计（见 §1.5） |

### 1.5 等保三级配套：审计日志、限流与 RBAC（补充）
* **操作审计：** 所有涉及医疗数据/资金/资质的写操作（接诊、开方、审方、提现、解密查看档案）必须落 `audit_logs`，记录操作者、动作、对象、IP、时间，长期留存备查。
* **限流防刷：** 登录、短信、支付下单等接口基于 Redis 做 IP/用户维度限流（如登录 5 次/分钟），超限返回 429。
* **RBAC：** 角色 `patient / doctor / pharmacist / admin / finance`，JWT 内置 `role`，接口用依赖注入做角色校验；管理后台按角色显示菜单。

## 二、 子系统 1：核心业务 API 引擎 (Core API Engine)

该子系统是整个平台的“中枢神经”，负责处理高并发的业务请求、状态流转以及微信生态的联动。面向对象为系统底层，不提供独立前端 UI，通过 RESTful API 供前端小程序调用。

### 2.1 微信支付订单状态机（核心合规模块）
系统必须实现严格的**封闭式订单状态机**，严禁出现状态逆流或非法跳转。

* **状态定义：**
    * `0: PENDING` (待支付)
    * `1: WAITING` (候诊中/排队中)
    * `2: CONSULTING` (问诊中/通话中)
    * `3: AUDITING` (待药师审核处方)
    * `4: REJECTED` (药师审核驳回)
    * `5: PRESCRIBED` (药师审核通过/已开方)
    * `6: FINISHED` (问诊完成)
    * `7: REFUNDED` (已退款)
    * `9: CANCELLED` (已取消)
* **核心流转逻辑：**
    * `0 -> 1`：用户支付成功（由微信支付回调 Webhook 触发）。
    * `0 -> 9`：用户主动取消或 15 分钟超时未支付自动取消。
    * `1 -> 2`：医生在工作台点击“立即接诊”。
    * `2 -> 3`：医生提交开具的电子处方，进入药师审核流。
    * `2 -> 6`：医生未开药直接结束问诊，或因无异常直接完成。
    * `3 -> 5`：药师审核通过，生成合法数字签名。
    * `3 -> 4`：药师认为用药不当驳回处方，通知医生重新修改。
    * `4 -> 3`：医生修改处方后重新提交审核。
    * `5 -> 6`：患者支付药费完成，或处方流转到院外。
    * `1/2 -> 7`：因网络故障、医生未接诊等原因触发的退款流程。
* **技术需求：** * 数据库订单表必须对 `status` 字段建立索引。
    * 微信支付回调必须实现**幂等性处理**。收到回调后，先使用 `SELECT ... FOR UPDATE` 锁住订单行，校验当前状态是否为 `0: 待支付`，确认后再变更为 `1: 候诊中`。

### 2.2 WebSocket 信令调度中心（双端实时交互）
负责双端的在线状态维系、被动唤醒呼叫、通话状态同步。
* **核心功能流（以医生呼叫患者为例）：**
    1. 医生在医生端小程序点击“立即接诊”，向后端发送 `INIT_CALL` 请求。
    2. 信令中心收到请求，校验医生状态后，向 Redis 写入一条诊室状态记录 `room:status:[room_id] = "calling"`。
    3. 信令中心通过 WebSocket 向**患者端**长连接推送一条 `CALL_INVITE` 格式信令，携带参数：`room_id`, `doctor_name`, `doctor_avatar`。
    4. 患者端点击接听，发送 `CALL_ANSWER` 信令；信令中心将 Redis 中房间状态改为 `"connected"`，并向双端广播 `START_STREAM` 信号。
* **断线重连（弱网机制）：** 维持心跳包（Ping/Pong）间隔为 4 秒。若连续 3 次未收到心跳，判定为断线。后端不立即销毁房间，保留状态 30 秒（断线保护期），等待小程序断线重连。超时则触发系统挂断。

### 2.3 三方 RTC 密钥与音视频 Token 计算安全
* **技术隔离：** 小程序端严禁保存三方视频服务商（如腾讯云 TRTC）的 `SDKAppID` 和 `SecretKey`。
* **动态计算：** 当前端触发进入音视频诊室前，必须调用后端的 `/api/v1/rtc/user-sig` 接口。后端读取当前登录态的 `user_id` 和传入的 `room_id`，在服务器端使用 HMAC-SHA256 算法计算出动态密钥 `UserSig`（Token），并设置有效期为 120 分钟，返回给前端。

### 2.4 用户鉴权与医疗隐私保护
* 系统采用 **JWT (JSON Web Token)** 进行无状态鉴权。Token 签名使用强密钥，存储于小程序的 `Storage` 中。
* **隐私去隐私化处理：** 后端返回的所有患者列表、病历记录中，身份证号、手机号、真实姓名必须在后端进行脱敏处理（如：张*三，138****1234）。只有进入真实音视频诊室的医生，才有权解密查看就诊人的完整电子健康档案。

### 2.5 就诊人实名认证与知情同意存证（补充，互联网诊疗准入硬性要求）
* **实名认证：** 添加就诊人须校验「姓名 + 身份证号」（接三方实名核验或公安二要素），身份证/手机号**加密存储**（非仅脱敏展示）。未实名的就诊人不得进入支付与问诊流。
* **知情同意存证：** 首次问诊前，患者签署《互联网诊疗知情同意书》《隐私政策》《医疗风险告知》，后端记录签署时间戳与协议版本快照（`consents` 表），供卫健委核查；未签署的订单禁止进入支付/音视频环节。
* **监护人代办：** 为儿童等无身份证场景预留监护人关系字段与代办授权。

## 三、 子系统 2：医生端工作台后台 (Doctor Web/App Backend)

该子系统为医生和审方药师提供业务操作的支撑。虽然移动端以微信小程序形式呈现，但后端需要为其提供完整的 API 支撑，并预留未来电脑 Web 端的扩展接口。

### 3.1 医生排班与诊金动态配置
* **数据模型：** 设计“排班模板表”和“出诊号源表”。排班精确到：日期 + 时间段（如：`2026-06-15 09:00:00 - 09:30:00`） + 号源上限（如：5人）。
* **号源锁机制：** 当患者端选中某时间段准备挂号时，后端在 Redis 中执行 `DECRBY号源数`。若返回值小于 0，说明无号，触发事务回滚。

### 3.2 诊室接诊工作台
* 提供当前排队队列的实时查询 API。支持按照挂号成功的时间戳进行升序排列。
* **接诊锁定：** 当医生点击“立即接诊”时，后端将该就诊人状态从“排队中”修改为“就诊中”。同时，必须将该医生在系统内的出诊状态设为“忙碌”，此时接诊大厅停止为其推流新患者。

### 3.3 电子病历 (EMR) 书写与流转系统
* 标准字段必须包含：主诉（Chief Complaint）、现病史（History of Present Illness）、既往史、初步诊断（支持 ICD-10 疾病字典联想输入）、处理意见。
* **保存校验：** 医生提交病历时，后端必须做非空与字数校验（如：主诉不得少于 5 个字），防止医生敷衍开方。

### 3.4 药师处方审核与一键驳回机制
根据合规要求，处方必须经独立药师审核。
* **双盲分发：** 医生开出处方后，该处方自动进入“待药师审核队列”。系统采用轮询或随机分发机制，推送给当前在线的执业药师后台。
* **药师操作：** * **通过：** 触发合规网关进行电子签名，处方生效，通知患者缴费。
    * **驳回：** 药师必须选择或填写“驳回原因”（如：抗生素用量超标、诊断与用药不符）。处方状态变为 `4: 审核驳回`，通过微信模板消息实时通知医生端小程序，要求医生重新修改。

## 四、 子系统 3：运营总管理后台 (SaaS Admin Backend)

该子系统是医院行政、财务、药房以及逑贝科技运营团队的管理中枢。采用 Web 端呈现（如使用 Vue3 + Element Plus 配合 Python 后端）。

### 4.1 医生入驻、多点执业资质终审模块
* **功能描述：** 收集和审查医生上传的执业资产。
* **关键字段：** 姓名、身份证号、医师资格证编号、医师执业证编号（第一执业地点与多点执业地点）、职称级别、科室分类。
* **技术要求：** 支持高清图片（PDF/JPG）的上传与存储（接入阿里云 OSS/腾讯云 COS），并提供图片防盗链访问控制。

### 4.2 院内药房与核心药品字典管理
* **互联网医院核心限制：** 互联网医院**严禁开具麻醉药品、精神药品、放射性药品等特殊管理药品**。
* **药品字典防线：** 运营后台的药品管理模块，必须对所有上架药品强制标记“处方药/非处方药/特殊限售药”属性。若系统检测到医生尝试搜索和添加标记为“特殊限售”的药品时，后端 API 直接拦截并报错提示“合规限制，本院不支持开具该特殊药品”。

### 4.3 财务对账、渠道分账与提现审批系统
* **流水计费：** 每一笔挂号费和药费订单，系统自动拆分为：总金额、医院留存、医生分成、平台技术服务费。
* **提现状态机：** 医生端发起提现请求 -> 后端冻结可提现余额 -> 进入 Admin 后台“人工财务审核” -> 审核通过 -> 调用微信支付“商家转账到零钱”API 进行打款 -> 接收微信成功回调 -> 解冻并扣除余额，完成归档。

### 4.4 监管数据上报监控面板
* 可视化展示每日数据上报给天津卫健委监管平台的成功率、失败数、延迟时长。
* **容错报警：** 若某条数据因网络波动上报失败，后台必须提供“一键重新上报”按钮，并支持配置错误日志查询。

## 五·补、 子系统 5：订单履约与消息通知中心（补充）

前端有「送药上门」与「消息诊室」，后端需对应模块支撑订单闭环与跨端通知。

### 5补.1 订单履约（药品配送）
* 药费支付成功后订单进入履约：药房配药 → 发货 → 物流追踪 → 签收，状态时间轴回传患者端「订单与物流」页。
* 对接物流/三方药房接口，记录运单号、承运商；支持申请售后/退款（触发 `7: REFUNDED`）。

### 5补.2 消息通知中心
* **微信订阅消息/模板消息：** 挂号成功、医生呼叫、处方已开、药费待付、药师驳回、物流更新、退款结果等关键节点统一经通知中心下发。
* **站内消息：** 落库为会话/系统通知，供患者端「消息诊室」与未读红点拉取。
* 通知与主业务**解耦**（异步队列下发），失败可重试，避免阻塞主流程。

## 五、 子系统 4：合规对接网关 (Compliance Gateway)

该子系统是**确保逑贝互联网医院能够通过天津卫健委年审、合规合法执业的命门**。它独立运行，作为微服务网关，专门负责与外部国家机构及认证中心的加密数据通信。

### 5.1 天津市互联网诊疗服务监管平台数据实时对接网关
* **安全认证机制：** 与卫健委通信必须使用特定的 **AES-128-CBC 加密算法** 或 **国密 SM4 算法**（具体以天津卫健委发放的最新规范为准），并在 HTTP Header 中携带动态计算的 `Sign`（签名 = MD5(Timestamp + AppSecret + Data)）。
* **异步队列上报（核心抗高并发设计）：** 为了防止卫健委服务器偶发性宕机或响应慢导致你们自己的小程序卡死，**严禁将上报接口写在主业务代码中**。
    * **正确做法：** 医生端点“接诊”或“开方”时，主业务将数据处理完毕后，直接往消息队列（RabbitMQ / Redis Stream）投递一条消息，立即给前端返回成功。
    * **网关消费：** 合规对接网关作为独立的后台 Worker 进程，高频从队列中取出数据，异步调用卫健委接口进行上报。
    * **重试机制：** 若卫健委接口返回 502 或超时，网关自动将该任务放入“死信队列”，每隔 5 分钟、15 分钟、1 小时进行指数退避重试，直至上报成功。

### 5.2 CA 机构电子数字签名与国密加签网关
根据《电子签名法》，医疗处方必须具备法律效力。
* **电子证书托管：** 网关对接三方合规 CA 机构。当医生在系统实名认证通过后，CA 机构在云端为该医生颁发一张数字证书（包含私钥，由 CA 托管）。
* **数字处方加签流程：**
    1. 当处方被药师审核通过时，网关提取处方核心摘要（包含：医院编码、处方号、医生姓名、诊断、药品列表、时间戳）。
    2. 网关调用 CA 机构的 **“云端静默签名接口”**，传入该医生的证书标识 and 处方摘要。
    3. CA 服务器使用国密 **SM2 算法** 对处方数据进行数字签名，返回加密哈希值（数字签名）。
    4. 网关后端利用 Python 的 `reportlab` 或 `pdfkit` 库，在服务器端动态生成标准处方 PDF 文件，将数字签名嵌入 PDF 属性中，同时在处方图片右下角盖上合法的**医院电子公章章及医生手写签章**。

### 5.3 医疗数据 15 年长期存储与备份策略
* 合规网关在音视频通话结束、处方生成完毕后，自动触发云端录制文件和处方 PDF 的归档任务。
* **归档逻辑：** 文件上传至云存储后，必须将存储类型变更为 **“归档存储（Archive）”**（成本仅为标准存储的 1/10），并设置生命周期策略（Lifecycle Policy）为 **5475 天（15年）内不允许删除**，完全满足国家卫健委对医疗档案长效保存的硬性法律检查。
