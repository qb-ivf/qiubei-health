# VS Code + AI 智能体开发微信小程序实操指南

本指南旨在指导研发团队如何利用 **VS Code** 作为核心代码编辑器，配合 **Gemini 3.1 Pro / Claude Code** 智能体进行高频代码生成，并以**微信开发者工具**作为最终预览与调试终端，实现快速迭代。

---

## 🛠️ 第一阶段：搭建环境与“双剑合璧”工作流

我们的核心策略是：**用 VS Code 写代码、用 AI 进补、用微信开发者工具看预览和调试。**

### 1. 准备小程序源码目录
1. 打开**微信开发者工具**，点击“加号”新建项目。
2. 选择一个空的本地目录，填入天津逑贝互联网医院小程序的 `AppID`（测试阶段可先选测试号）。
3. 后端开发语言选择 **JavaScript/TypeScript**，样式选择 **Less/Sass**，模板选择 **“不使用模板”** 或 **“基础模板”**。
4. 创建成功后，将微信开发者工具最小化，仅将其用作**真机预览、真机调试与代码提审终端**。

### 2. 配置 VS Code 开发环境
用 VS Code 打开刚刚创建的那个项目目录，安装以下核心插件：
* **WeChat Snippet：** 提供微信小程序代码辅助提示。
* **WXML / Wxml Webpack：** 提供 WXML 和 WXSS 的语法高亮及自动补全。
* **Easy Less 或 Easy Sass：** 编写 Less/Sass 时，插件会自动在同级目录下编译生成小程序需要的 `.wxss` 文件。

### 3. 配置 AI 智能体
* 使用 VS Code 插件生态（如 Roo Code、Continue 等），将 **Gemini 3.1 Pro** 作为底层大模型。利用其长上下文（200万 Token）特性，可将小程序整套 PRD 及外部接口文档同时作为上下文喂给它。
* 确保已赋予 AI 助手当前项目 Workspace 目录的读写权限。

## 🚀 第二阶段：AI 自动化开发的“四步战术”

不要试图让 AI 一下子把 10 个页面全写完。按照**“主包基础设施 ➡️ 单页切图 ➡️ 信令与全局状态机 ➡️ 核心 Bug 攻坚”**的顺序推进。

### 🛰️ 步战术 1：初始化全局配置（主包与分包架构）
**提示词示例（发送给 Gemini）：**
> “我是微信小程序开发者。请根据以下分包需求，帮我编写项目的 `app.json` 配置文件。要求：主包包含首页和个人中心，问诊流放入独立分包 `subpackages/consult`，医生工作台放入 `subpackages/doctor`。并开启 Less 编译支持。请直接生成完整的配置文件代码。”

### 🎨 步战术 2：单页像素级切图（以患者端首页为例）
**提示词示例（发送给 Gemini）：**
> “请帮我编写主包中的 `pages/index/index.wxml` 和 `index.less` 代码。需求如下：
> [此处粘贴 mini_program_prd.md 中的‘页面 1：小程序首页’需求]
> 视觉要求：主色调医疗蓝 #2A76F4，核心功能卡片必须有 12px 圆角和微弱阴影，整体符合适老化大字号规范。请给出完整的布局和样式代码。”
*保存代码后，切换到微信开发者工具，查看右侧模拟器渲染效果。如有细节不满意，直接在 VS Code 中追加提示词微调（如：“金刚区卡片太挤了，加个 padding”）。*

### 🔌 步战术 3：攻坚核心逻辑（WebSocket 信令与 RTC 绑定）
**提示词示例（发送给 Gemini）：**
> “我正在开发全屏双向视频诊室页面（页面 5）。请帮我编写 `video-room.js` 的核心逻辑。
> 要求：
> 1. 在 `onLoad` 时从全局获取 WebSocket 实例并初始化监听。
> 2. 当收到服务器下发 `type: "CALL_INVITE"` 信令时，触发路由跳转。
> 3. 页面内需管理 `isSwapped` 状态，实现点击小窗时，无缝对调大窗 `<live-player>` 和小窗 `<live-pusher>` 的布局层级（Z-Index）、宽高和样式。
> 4. 当监听到 WebSocket 收到 `CALL_FINISHED` 信号时，自动销毁组件流并跳转至处方页。
> 请给出完整的 JavaScript/TypeScript 逻辑代码，并加入详尽的注释。”

### ⌨️ 步战术 4：攻坚医生端（防软键盘遮挡 Bug）
**提示词示例（发送给 Gemini）：**
> “在医生端视频开方页面（页面 D2），上部 40vh 是视频区，下部 60vh 是 Scroll-View 表单作业区。请帮我编写 WXML 和 JS 逻辑，利用微信的 `bindfocus` 监听键盘弹起高度，动态改变下部区域的 `padding-bottom`。必须强制设置 `adjust-position="false"`，确保手机软键盘弹起时，上部视频区绝对不被往上推。请给出完美的防遮挡前端解决方案。”

## 🛠️ 第三阶段：真机调试与联调（脱离模拟器）

当前端逻辑和后端的 Python FastAPI 接口初步完成后，必须进行双端真机对打测试。

### 1. 解开域名限制（局域网本地联调）
* 在微信开发者工具右上角点击 **【详情】->【本地设置】**。
* 勾选 **“不校验合法域名、web-view（业务域名）、TLS版本以及HTTPS证书”**。
* 此时，前端小程序可以直接向本地局域网的后端 IP 发起请求（例如：`http://192.168.X.X:8000`），无需部署线上 HTTPS 证书即可联调。

### 2. 双真机对打演练
* 准备两部手机，一部登录患者端，一部登录医生端（利用开发者工具的“真机调试”功能分别扫码进入）。
* 让医生端点击接诊，测试 WebSocket 是否能秒级唤醒患者端，测试腾讯云 TRTC 画面是否能顺利拉取。

### 3. 利用 AI 智能体捉虫（Debug）
联调时如果真机控制台报错（例如：`TypeError: Cannot read property 'setData' of undefined`），直接将报错信息和上下文 JS 代码丢给 Gemini：
> “在真机重连时报了以下错误：[粘贴报错信息]。这是我的核心 JavaScript 代码：[粘贴代码]。请帮我分析原因并给出修复方案。”
