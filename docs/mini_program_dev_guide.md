# VS Code + AI 智能体开发微信小程序实操指南

本指南旨在指导研发团队如何利用 **VS Code** 作为核心代码编辑器，配合 **Gemini 3.1 Pro / Claude Code** 智能体进行高频代码生成，并以**微信开发者工具**作为最终预览与调试终端，实现快速迭代。

> ⚠️ **架构前提（与 FRD §1.1 对齐）：** 患者端「逑贝健康」与医生端「逑贝医生端」是**两个物理隔离的独立小程序**（独立 AppID、独立 codebase、独立审核发布），仅通过中央服务端通信。因此下文的"建项目""写 `app.json`""提审"等动作，**患者端与医生端各做一遍**；建议在仓库中用 `miniprogram-patient/` 与 `miniprogram-doctor/` 两个目录分别承载。

---

## 📋 实施阶段总览

下表是本指南各阶段的速查地图，建议按顺序推进，每个阶段都有明确的「进入条件 → 产出物 → 完成标准」。

| 阶段 | 核心目标 | 主要产出物 | 完成标准（验收点） |
| :--- | :--- | :--- | :--- |
| **一、搭建环境与工作流** | 打通 VS Code 写码、AI 补码、微信工具预览的「三位一体」链路 | 可运行的空项目骨架、装好插件的 VS Code、配好的 AI 智能体 | 改 Less 能自动生成 wxss，模拟器能跑起空白页 |
| **二、AI 自动化开发（五步战术）** | 从主包配置到核心逻辑，分批让 AI 产出**两端**（患者端 10 页 + 医生端 3 页）代码 | 两端各自的 `app.json`、各页 WXML/Less/JS、合规闸门、信令状态机、防遮挡与开方、微信支付 | 模拟器中各页面渲染正常、路由可跳转、支付与信令链路自洽 |
| **三、真机调试与联调** | 脱离模拟器，前后端双端真机对打 | 局域网联调通过的前后端、修复后的真机 Bug | 医生端接诊能秒级唤醒患者端、TRTC 双向画面正常 |
| **四、提审、上线与合规**（补充） | 通过微信审核与医疗资质核验，灰度发布 | 体验版/审核版、合规材料、线上 HTTPS 域名 | 微信审核通过、互联网医院资质与监管接口对接就绪 |

> 💡 **节奏建议：** 阶段一为一次性投入（约半天）；阶段二是主战场，按「步战术」逐页迭代；阶段三需前后端同步就绪后启动；阶段四涉及行政与资质，应在阶段一就**并行启动**（见文末团队分工），避免成为上线瓶颈。

## 🛠️ 第一阶段：搭建环境与“双剑合璧”工作流

> **🎯 本阶段目标：** 一次性打通开发链路，让「写码 → 编译 → 预览」实现零摩擦闭环。
> **✅ 进入条件：** 已申请到小程序 AppID（或测试号）、本机已安装 Node.js 与微信开发者工具。
> **📦 产出物：** 空项目骨架、装好插件的 VS Code、配置完成的 AI 智能体。

我们的核心策略是：**用 VS Code 写代码、用 AI 进补、用微信开发者工具看预览和调试。**

### 1. 准备小程序源码目录（两个独立项目）
> 患者端与医生端是两个独立小程序，需在微信开发者工具里**各建一个项目**，对应两个不同的 AppID。流程相同，做两遍：
1. 打开**微信开发者工具**，点击“加号”新建项目。
2. 选择一个空的本地目录（建议 `miniprogram-patient/` 与 `miniprogram-doctor/`），分别填入患者端 / 医生端的 `AppID`（测试阶段可先各用一个测试号）。
3. 后端开发语言选择 **JavaScript/TypeScript**，样式选择 **Less/Sass**，模板选择 **“不使用模板”** 或 **“基础模板”**。
4. 创建成功后，将微信开发者工具最小化，仅将其用作**真机预览、真机调试与代码提审终端**。两个项目可在同一开发者工具中切换打开。

### 2. 配置 VS Code 开发环境
用 VS Code 打开刚刚创建的那个项目目录，安装以下核心插件：
* **WeChat Snippet：** 提供微信小程序代码辅助提示。
* **WXML / Wxml Webpack：** 提供 WXML 和 WXSS 的语法高亮及自动补全。
* **Easy Less 或 Easy Sass：** 编写 Less/Sass 时，插件会自动在同级目录下编译生成小程序需要的 `.wxss` 文件。

### 3. 配置 AI 智能体
* 使用 VS Code 插件生态（如 Roo Code、Continue 等），将 **Gemini 3.1 Pro** 作为底层大模型。利用其长上下文（200万 Token）特性，可将小程序整套 PRD 及外部接口文档同时作为上下文喂给它。
* 确保已赋予 AI 助手当前项目 Workspace 目录的读写权限。
* **喂料建议：** 把 `docs/mini_program_prd.md`（页面需求）和后端接口文档一并加入上下文，让 AI 生成的代码字段命名、路由参数与 PRD 保持一致，减少返工。

### 4. 初始化版本控制与协作规范
* 在项目根目录执行 `git init`，并新建 `.gitignore`，**务必忽略** `miniprogram_npm/`、`node_modules/`、微信开发者工具生成的 `project.private.config.json` 等。
* 将 `.wxss`（由 Less/Sass 编译产出）是否纳入版本库与团队约定统一：建议**忽略编译产物、只提交源 `.less`**，避免冲突。
* 约定分支策略（如 `main` 保护、功能分支开发），AI 大批量改码后**先本地预览验证再提交**，防止把跑不通的代码推上去。

> **🔍 完成自检：** 修改任意 `.less` 文件保存后能在同级自动生成 `.wxss`；微信开发者工具模拟器能加载出空白首页且控制台无致命报错——满足即可进入第二阶段。

## 🚀 第二阶段：AI 自动化开发的“五步战术”

> **🎯 本阶段目标：** 用「小批量、可验证」的方式让 AI 逐步产出全部页面与核心逻辑，每一步都在模拟器中即时验收。
> **✅ 进入条件：** 第一阶段链路已打通，PRD 已作为 AI 上下文。
> **📦 产出物：** `app.json` 分包+tabBar 配置、全局登录守卫、10 个页面的 `WXML/Less/JS`、视频信令状态机、医生端防遮挡与开方组件、微信支付闭环。

不要试图让 AI 一下子把 10 个页面全写完。按照**“主包基础设施 ➡️ 单页切图 ➡️ 信令与全局状态机 ➡️ 医生端核心 Bug 攻坚 ➡️ 支付闭环”**的顺序推进。**每完成一步即在微信开发者工具中验证，再进行下一步**，避免错误累积难以定位。

### 🛰️ 步战术 1：初始化全局配置（两端各自的主包与分包 + 全局状态机）
> 两个小程序各写一份 `app.json`。下面分别给患者端、医生端的提示词。

**患者端 `app.json`（提示词）：**
> “我是微信小程序开发者。请帮我编写**患者端**小程序的 `app.json`：主包含首页(页面1)和个人中心(页面7)，问诊流(页面2/3/4/5/6)放入分包 `subpackages/consult`。开启 Less 编译。配置全局 4 个底部 Tab：[首页]、[科室挂号]、[消息诊室]、[我的]（含选中/未选中图标占位），并预留分包预下载 `preloadRule`。请直接生成完整配置文件代码。”

**医生端 `app.json`（提示词）：**
> “请帮我编写**医生端**小程序的 `app.json`：主包含接诊大厅(D1)和工作台(D3)，视频开方页(D2)放入分包 `subpackages/consult`。开启 Less 编译，配置医生端底部 Tab（如 [接诊大厅]、[我的]）。请直接生成完整配置文件代码。”

**补充：同步搭好「全局状态机」地基（对应 FRD §1.2、§二）。** 主包配置完成后，立即让 AI 在 `app.js` 中落地全局机制，后续页面才能直接复用：
> “请在 `app.js` 的 `onLaunch` 中封装：① 全局**登录路由守卫**——本地 Storage 无有效 Token 的用户进入需登录页面时，硬性弹出微信手机号一键授权登录弹窗（医生端额外校验后端白名单）；② 全局 `globalData` 状态（当前就诊人、Token、WebSocket 实例占位）；③ **准入闸门工具**：进入支付/问诊前校验"就诊人已实名 + 知情同意已签署"，未满足则拦截并引导。再单独封装一个**音视频权限校验工具函数**，进入视频相关页面前调用 `wx.getSetting` 校验 `scope.camera` 与 `scope.record`，被拒时返回需引导设置的标记。请给出完整代码与注释。”

### 🎨 步战术 2：单页像素级切图（以患者端首页为例）
**提示词示例（发送给 Gemini）：**
> “请帮我编写主包中的 `pages/index/index.wxml` 和 `index.less` 代码。需求如下：
> [此处粘贴 mini_program_prd.md 中的‘页面 1：小程序首页’需求]
> 视觉要求：品牌主色医疗蓝 #2A76F4（导航/主按钮），**金刚区"视频问诊"卡片用强调绿 #07C160**（对齐 FRD §1.4）；核心功能卡片必须有 12px 圆角和微弱阴影 `0 2px 8px rgba(0,0,0,.06)`，正文字号 ≥ 32rpx、按钮高度 ≥ 88rpx，符合适老化无障碍规范。请给出完整的布局和样式代码。”
*保存代码后，切换到微信开发者工具，查看右侧模拟器渲染效果。如有细节不满意，直接在 VS Code 中追加提示词微调（如：“金刚区卡片太挤了，加个 padding”）。*

### 🔌 步战术 3：攻坚核心逻辑（WebSocket 信令与 RTC 绑定）
**提示词示例（发送给 Gemini）：**
> “我正在开发全屏双向视频诊室页面（页面 5）。请帮我编写 `video-room.js` 的核心逻辑。
> 要求：
> 1. 在 `onLoad` 时从全局获取 WebSocket 实例并初始化监听。
> 2. 当收到服务器下发 `type: "CALL_INVITE"` 信令时，触发路由跳转。
> 3. 页面内需管理 `isSwapped` 状态，实现点击小窗时，无缝对调大窗 `<live-player>` 和小窗 `<live-pusher>` 的布局层级（Z-Index）、宽高和样式。
> 4. 当监听到 WebSocket 收到 `CALL_FINISHED` 信号时，自动销毁组件流并跳转至处方页。
> 请给出完整的 JavaScript/TypeScript 逻辑代码，并加入详尽的注释。”

**📡 WebSocket 信令状态机速查表（与后端务必字字对齐）：** FRD 的呼叫链路共涉及 6 类信令，前端需在 `app.js` 全局监听器中分发，切勿只处理 `CALL_INVITE`/`CALL_FINISHED`：

| 信令 `type` | 方向 | 前端动作 |
| :--- | :--- | :--- |
| `INIT_CALL` | 医生端 → 服务端 | 医生在 D1 点击接诊，发起呼叫 |
| `CALL_INVITE` | 服务端 → 患者端 | **任意页面**立即 `navigateTo` 拉起页面4 接听页 |
| `CALL_REJECT` | 患者端 → 服务端 | 页面4 点"拒绝"，清响铃并 `navigateBack` |
| `CALL_ANSWER` | 患者端 → 服务端 | 页面4 点"接听"，`redirectTo` 进页面5（不留栈） |
| `CALL_FINISHED` | 服务端/医生端 → 患者端 | 页面5 销毁流，跳页面6 生成处方 |

> ⚠️ **全局 vs 页面：** `CALL_INVITE` 的监听必须挂在 `app.js` 全局长连接里（用户可能停在任意页），而非页面5 内部——这是新手易错点。页面5 只负责接管已建立的房间与 `CALL_FINISHED`。

### ⌨️ 步战术 4：攻坚医生端（防软键盘遮挡 Bug + 开方组件）
**提示词示例（发送给 Gemini）：**
> “在医生端视频开方页面（页面 D2），上部 40vh 是视频区，下部 60vh 是 Scroll-View 表单作业区。请帮我编写 WXML 和 JS 逻辑，利用微信的 `bindfocus` 监听键盘弹起高度，动态改变下部区域的 `padding-bottom`。必须强制设置 `adjust-position="false"`，确保手机软键盘弹起时，上部视频区绝对不被往上推。请给出完美的防遮挡前端解决方案。”

**补充：开方作业面板的三个 Tab（对应 FRD 页面 D2）。** 防遮挡只是骨架，开方核心交互也要让 AI 一并产出：
> “请继续完善页面 D2 下部作业面板的三个 Tab：① `[患者主诉]`——文本展示 + 既往病例九宫格，点击触发 `wx.previewImage` 全屏预览；② `[书写病历]`——[现病史]/[初步诊断]/[医嘱] 三个 `textarea`；③ `[开具处方]`——顶部药名拼音首字母搜索框带实时下拉联想，选中药品以卡片塞入‘已开药品列表’，每张卡片含数量 Stepper 组件和用法用量 Picker 组件。底部常驻‘数字签名并发送处方’按钮，点击触发 CA 签名 loading，成功后清栈强退回 D1。请给出完整 WXML/JS 代码。”

### 💳 步战术 5（补充）：微信支付闭环（页面3 挂号费 / 页面6 药费）
FRD 中页面3 与页面6 都走微信原生支付，是 10 个页面里**唯二涉及资金**的环节，必须单独攻坚并真机验证。
**提示词示例（发送给 Gemini）：**
> “请帮我封装微信小程序支付的前端通用方法 `requestPay(orderId)`：① 向后端下单接口请求并拿到支付五元组（`timeStamp`、`nonceStr`、`package`、`signType`、`paySign`）；② 调用 `wx.requestPayment`；③ `success` 回调里执行业务跳转（页面3 支付成功 → Toast 后 `wx.switchTab` 回首页并挂全局置顶黄色提示条；页面6 支付成功 → 跳页面7 并把订单归档为‘待药房配送’）；④ `fail` 回调区分用户主动取消与真实失败。请给出完整代码与防重复支付处理。”

> 📌 **步战术之外的剩余页面：** 上述 5 个步战术给出的是"硬骨头"范式。其余页面（页面2 列表、页面7/D3 个人中心、页面D1 接诊大厅开关状态机）复杂度较低，套用步战术2 的"切图 + 粘贴 FRD 需求"模式逐页产出即可，按上方〔十页面开发地图〕的排期推进，确保 10 页无遗漏。

## 🛠️ 第三阶段：真机调试与联调（脱离模拟器）

> **🎯 本阶段目标：** 在真实手机与局域网环境下验证前后端协同，重点攻克模拟器无法复现的音视频、WebSocket、软键盘等真机问题。
> **✅ 进入条件：** 前端页面在模拟器跑通，后端 FastAPI 核心接口（登录、医生列表、信令、TRTC 鉴权）已可本地启动。
> **📦 产出物：** 局域网联调通过的前后端、修复后的真机 Bug 清单。

当前端逻辑和后端的 Python FastAPI 接口初步完成后，必须进行双端真机对打测试。

### 1. 解开域名限制（局域网本地联调）
* 在微信开发者工具右上角点击 **【详情】->【本地设置】**。
* 勾选 **“不校验合法域名、web-view（业务域名）、TLS版本以及HTTPS证书”**。
* 此时，前端小程序可以直接向本地局域网的后端 IP 发起请求（例如：`http://192.168.X.X:8000`），无需部署线上 HTTPS 证书即可联调。

### 2. 双真机对打演练
* 准备两部手机，一部登录患者端，一部登录医生端（利用开发者工具的“真机调试”功能分别扫码进入）。
* 让医生端点击接诊，测试 WebSocket 是否能秒级唤醒患者端，测试腾讯云 TRTC 画面是否能顺利拉取。

### 3. 利用 AI 智能体捉虫（Debug）
联调时如果真机控制台报错（例如：`TypeError: Cannot read property 'setData' of undefined`），直接将报错信息和上下文 JS 代码丢给 Gemini：
> “在真机重连时报了以下错误：[粘贴报错信息]。这是我的核心 JavaScript 代码：[粘贴代码]。请帮我分析原因并给出修复方案。”

### 4. 端到端验收清单（对照 FRD 全链路）
真机联调不是"能连上"就算过，需逐项跑通 FRD 定义的关键链路：
- [ ] **登录守卫**：未登录访问页面2/3/D1/D2 时弹一键授权登录。
- [ ] **音视频授权**：进入页面4/5/D2 前 `wx.getSetting` 校验，拒绝时展示引导页。
- [ ] **挂号支付**：页面3 `wx.requestPayment` 成功 → 回首页挂黄色置顶提示条。
- [ ] **呼叫唤醒**：医生 `INIT_CALL` → 患者**任意页面**被 `CALL_INVITE` 拉起页面4。
- [ ] **双向视频**：页面5 `<live-player>`/`<live-pusher>` 双画面正常、大小窗对调无误。
- [ ] **挂断闭环**：医生开方 → `CALL_FINISHED` → 患者跳页面6 看到带 CA 红章处方。
- [ ] **药费支付**：页面6 支付成功 → 订单归档"待药房配送"。
- [ ] **防遮挡**：D2 软键盘弹起，上部 40vh 视频区纹丝不动。

## 🚦 第四阶段：提审、上线与合规（补充）

> **🎯 本阶段目标：** 把联调通过的小程序推向线上，重点不是技术、而是**医疗资质与微信审核红线**——这是医疗类小程序最容易卡住的环节，须尽早并行准备。
> **✅ 进入条件：** 真机联调通过；天津主体营业执照、《互联网医院牌照》及《互联网药品信息服务资格证》等资质材料已在办理或就绪。
> **📦 产出物：** 体验版/审核版、上线的 HTTPS 业务域名与 WSS 信令域名、卫健委监管接口对接、CA 电子签名链路。

### 1. 域名、备案与合法域名配置
* 后端从联调用的局域网 IP **切换为线上 HTTPS 域名**，配置 SSL 证书；信令服务使用 **WSS**。
* 在微信公众平台「开发管理 → 服务器域名」中登记 `request`、`socket`、`uploadFile`、`downloadFile` 合法域名，并提前完成 ICP 备案。
* **上线前务必关闭**第三阶段勾选的「不校验合法域名」开关，否则真机非调试环境会请求失败。

### 2. 医疗资质与合规接入（最易卡审，须提前启动）
* **类目资质：** 微信医疗类目需提交《互联网医院牌照》《医疗机构执业许可证》等，建议在阶段一即由研发 C 并行办理。
* **卫健委监管上报：** 对接天津市卫健委监管平台，按规范（AES-128-CBC + MD5-Sign）实时上报医生资质、问诊记录、开药明细。
* **CA 电子签名：** 完成国密合规 CA 机构对接，确保处方加盖 SM2 数字证书红章后方可流转支付。

### 3. 提审与灰度发布（两端各提一次）
* **患者端、医生端是两个独立小程序，需分别上传、分别提交审核**，各自维护版本号与体验版。
* 在微信开发者工具中切换到对应项目「上传」代码生成**体验版**，邀请团队与试点医生灰度测试。
* 整理审核所需的资质截图、功能录屏与测试账号，提交审核；医疗类目通常需 1～7 个工作日且易被打回，预留缓冲。医生端常因"未明确执业资质展示"被打回，需重点准备。
* 审核通过后先**小流量灰度发布**，监控支付状态机、TRTC 接通率与监管上报成功率，稳定后再全量。

> **🔍 完成自检：** 微信审核通过、线上域名 HTTPS/WSS 请求正常、首单真实问诊可完成「接诊→开方→CA 签章→支付→监管上报」全链路——满足即正式上线。
